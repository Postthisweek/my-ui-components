Best Practices for Accessible Buttons

Primary Buttons

Primary buttons are the main call-to-action on a page. They should be visually distinct and easy to identify.

  • Use for the main action on a page (e.g., "Submit", "Save", "Continue")
  • Limit to one primary button per section to avoid confusion
  • Avoid overuse, if everything is a primary button, users may struggle to find the main action

Secondary Buttons

Secondary buttons are used for less important actions. They should be visually distinct but not overpowering.

  • Use secondary buttons for actions that are not the primary focus
  • Use for actions that are important but not the main focus (e.g., "Cancel", "Back")
  • Ensure they are visually distinct from primary buttons
  • Should be visually distinct but not overpowering

Danger Buttons

Danger buttons are used for actions that may have negative consequences. They should be visually distinct and clearly labeled.

  • Use danger buttons for actions that may have negative consequences (e.g., "Delete", "Remove")
  • Ensure they are visually distinct from primary and secondary buttons
  • Use for actions that may have negative consequences (e.g., "Delete", "Remove")

Outline Buttons

Outline buttons are used for actions that are less important than primary buttons. They should be visually distinct but not overpowering.

  • Use outline buttons for actions that are less important than primary buttons
  • Ensure they are visually distinct from primary and secondary buttons
  • Avoid using outline buttons in low-contrast themes or without proper padding‚Äîthey may become hard to perceive

Text-Only Buttons

Text-only buttons are used for actions that are less important than primary buttons. They should be visually distinct but not overpowering.

  • Use text-only buttons for actions that are less important than primary buttons
  • Ensure they are visually distinct from primary and secondary buttons
  • Used for subtle actions where a button-like appearance isn‚Äôt necessary (e.g., "Learn More")
  • Often found in links that act like buttons but don‚Äôt require emphasis
  • Useful in minimalist designs that reduce UI clutter
  • Should not be used for critical actions like "Submit" or "Delete", these need more visibility

Icon buttons are used for actions that are less important than primary buttons. They should be visually distinct but not overpowering.

Disabled Buttons

Disabled buttons are used for actions that are not currently available. They should be visually distinct and clearly labeled.

  • Use disabled buttons for actions that are not currently available
  • Ensure they are visually distinct from enabled buttons
  • Use for actions that are not currently available (e.g., "Submit" when a form is incomplete)
  • Should be visually distinct and clearly labeled
  • Ensure they are visually distinct from enabled buttons
  • Use a lighter color or a different style to indicate the disabled state
  • Provide a tooltip or message explaining why the button is disabled

Interactive Buttons

Interactive buttons are used for actions that require user interaction. They should be visually distinct and clearly labeled.

  • Use interactive buttons for actions that require user interaction
  • Ensure they are visually distinct from other elements
  • Avoid overcomplexity, if the action is essential, ensure users can predict the button's behaviour
  • Used for actions that trigger a change (e.g., opening modals, toggling a feature, dynamic events)
  • Found in interactive elements, such as expanding sections or triggering alerts
  • Essential for user-driven experiences that respond to input beyond navigation

Best Practices for All Buttons

Regardless of the button type, here are some best practices to follow:

  • Use clear and concise text that describes the action
  • Ensure buttons are large enough to be easily clickable
  • Use consistent styling across all buttons

  • Use ARIA roles and properties to enhance accessibility
  • Ensure the button is visually distinct from other elements
  • Should have focus states and keyboard accessibility (Tab, Enter, Space)
  • Provide high contrast, improving visibility for users with low vision

No Pre-selection vs. Pre-selection Options

Radio buttons should not have a pre-selected option unless it is necessary for the user to make a choice. If a pre-selection is made, ensure that the user can easily change their selection.

No Pre-selection

When no option is pre-selected, users are required to make a choice before proceeding. This can be beneficial in scenarios where the user must actively engage with the content.

  • Implied Endorsement: Pre-selecting an option can inadvertently suggest that this choice is the "default" or preferred option, potentially biasing the user's decision without them actively considering all the choices
  • False Submissions: Users might quickly scan the form and submit it without realizing a radio button was already selected, leading to unintended data
  • Clarity of Choice: Leaving all options unselected makes it clear that the user needs to make an active choice
Pre-selection

Pre-selecting an option can be useful in scenarios where a default choice is common or expected. However, it is essential to ensure that users are aware of the pre-selection and can easily change it if desired.

  • Default Choice: In some cases, a default choice may be appropriate, such as when a specific option is the most commonly selected
  • Guidance: Pre-selection can guide users toward a recommended action, especially in complex forms
  • Clarity of Choice: Ensure that users are aware of the pre-selection and can easily change it if desired
  • Accessibility: Ensure that the pre-selected option is clearly indicated and that users can easily change their selection

Best Practices for Alerts

  • Use ARIA Roles & Attributes: role="alert" for assertive, immediate notifications (live updates); use role="status" for polite notifications that do not interrupt users and use aria-live="polite" or aria-live="assertive" to control how assistive technologies announce changes
  • Ensure Keyboard Accessibility: Allow users to dismiss alerts using the Esc key; ensure focus management (e.g., send focus to the alert when triggered) and provide logical tab order and aria-describedby for clarity
  • Responsive & Scalable Design: Use relative units (em, %, rem) instead of fixed pixels and implement flexbox or grid for adaptable layouts and consider contrast ratios for readability

Best Practices for Notifications

Notifications are essential for keeping users informed. Here are some best practices to ensure accessibility:

  • ARIA Live Regions: Use aria-live="polite" for non-critical notifications and aria-live="assertive" for urgent ones; use role attributes: role="alert" for error messages, role="status" for general updates
  • Keyboard Navigation: Ensure notifications can be dismissed with keyboard (usually Escape key), ensure focus management by moving focus to new critical notifications but provide a way to return to previous location
  • Colour Contrast: Minimum 4.5:1 for text against background (WCAG AA), ensure multiple indicators: combine colour with icons and text for universal understanding
  • Responsive Layouts: Stack notifications vertically on mobile, consider position (top-right on desktop, bottom on mobile)
  • Timing: Persistent for important notices, auto-dismiss after 5-8 seconds for others; ensure grouping: Bundle similar notifications to avoid overload, and enable persistence: critical notifications should remain until manually dismissed
  • Clear Language: Concise, actionable text; maintain priority levels: visual distinction between info, success, warning, and error. Include relevant buttons (e.g., "Undo", "View", "Dismiss")

Best Practices for Error Message & Summary

Effective error messages and summaries are crucial for user experience. Here are some best practices to ensure accessibility:

Error Messages (Field-Level Feedback)

  • Clear and Specific: Indicate what went wrong and how to fix it (e.g., "Invalid email format user(at)example.com") and avoid vague messages like "Something went wrong"
  • Visible and Screen Reader-Friendly: Place error messages near the input field they relate to. Use aria-describedby="error-message-id" to link error messages to inputs. Set aria-live="assertive" for critical errors that need immediate attention
  • Keyboard and Mouse Interactivity: Make errors focusable (tabindex="-1") so keyboard users can navigate to them. Allow users to dismiss messages with the Escape key or a close button
  • Consistent Styling: Use consistent colors, fonts, and icons for error messages. Ensure sufficient contrast between text and background (4.5:1 ratio)
  • Validation on Input: Provide real-time feedback as users type, indicating errors immediately rather than waiting for form submission

Error Summary (Global Feedback)

  • Comprehensive Overview: Provide a summary of all errors at the top of the form, allowing users to quickly identify and address issues
  • Screen Reader Compatibility: Use role="alert" for error summaries to ensure they are announced immediately. Use aria-live="assertive" for critical errors
  • Clear Instructions: Include instructions on how to resolve errors in the summary (e.g., "Please correct the highlighted fields")
  • Accessible Dismissal: Allow users to dismiss the error summary with a close button or by pressing the Escape key
  • Allow Quick Fix Navigation: Each error in the summary should be clickable, focusing the related field. Example: Clicking "Email is required" moves focus to the email input.

New Ideas & Theoretical Innovations

Explore new ideas and theoretical innovations in error message and summary design:

  • Dynamic Error Messaging: Use machine learning to adapt error messages based on user behavior and preferences
  • Contextual Help: Provide contextual help or tooltips that appear when users hover over error messages, offering additional guidance
  • Visual Cues: Use animations or visual cues (e.g., shaking input fields) to draw attention to errors without being disruptive
  • Gamification: Introduce gamified elements, like progress bars or badges, to encourage users to correct errors and complete forms
  • Collaborative Error Resolution: Allow users to share error messages with support teams or peers for collaborative troubleshooting
  • Multi-Sensory Feedback: Combine visual, auditory, and haptic feedback for error messages to cater to different user preferences
  • AI-Powered Error Prevention: Instead of showing errors after submission, predict mistakes using AI (e.g., autocomplete suggests valid formats before entry)
  • Adaptive Error Sensitivity: Users can customize how errors are shown (e.g., neurodivergent-friendly ‚Äúgentle‚Äù mode vs. urgent alerts)
  • Context-Aware Error Highlights: Dynamically adjust error placement based on screen size and user attention

Best Practices for Accordion

Accordions are useful for displaying collapsible content sections. Here are some best practices to ensure accessibility:

  • Ensure that accordion has clear accessible labels for each section for user to understand the content hierarchy when using screen readers
  • The focus state is well-defined, allowing user to easily identify which section they are interacting with when navigating via keyboard
  • The accordion expands and collapses sections using standard ARIA roles and properties, ensuring compatibility with assistive technologies
  • Keyboard shortcuts allow should open or close sections (e.g., pressing Enter or Space on the header of an accordion item)
  • The state of the accordion (expanded or collapsed) is announced by screen readers when toggling. For instance, it might say, "Introduction, collapsed" or "Introduction, expanded"
  • Responsive Design: Styles ensure the accordion adapts well to various screen sizes using a fluid layout
  • Speech Recognition Tools: Clear, hierarchical semantic structure allows smooth interaction for users relying on voice commands

Accordion Implementation Practices

The practice for only one section of accordions is expanded at a time is known as a single-open accordion. It helps streamline the user experience by reducing the cognitive load and minimizing the amount of visible content at once, which can be especially beneficial for screen reader users.

Then, there is a multi-open accordion, where multiple sections can be expanded simultaneously.

This design isn't inherently wrong. It depends on the context and user needs:

  • Single-Open Accordion: Works best when the content of each section is lengthy or when the user is expected to focus on one topic at a time. It prevents overwhelming the user by showing too much information
  • Multi-Open Accordion: Suitable for scenarios where users might want to compare information across sections or where the content is relatively concise. For example, a FAQ section where users might want to see multiple questions and answers at once. However, it can lead to cognitive overload if too many sections are open at once
  • Best Practice: Consider user needs and context. If using a multi-open accordion, ensure that the design allows users to easily collapse sections they no longer need
  • Testing: Always test with real users to see which design works best for your specific audience

Best Practices for Modal Dialogs

Modal dialogs are useful for displaying important information or actions. Here are some best practices to ensure accessibility:

  • Must be properly labelled with ARIA attributes
  • Duration Guidelines: Should appear immediately when triggered (no animation delay); should remain visible until user action (no auto-dismissal); animations should be brief (200-300ms) and optional for reduced motion
  • Must trap and manage keyboard focus
  • Must provide keyboard-accessible controls
  • Must be announced to screen readers
  • Must have proper backdrop interaction
  • Must support escape key dismissal

When to use Modals

Modals are best used for:

  • Critical information that requires immediate user attention
  • Actions that need to be completed before proceeding (e.g., confirmation dialogs)
  • Forms or inputs that are contextually relevant to the current task
  • Alerts or notifications that require user acknowledgment
  • Content that is secondary to the main task but still important (e.g., help or support)

When NOT to use Modals

Modals should be avoided for:

  • Non-critical information that can be displayed inline
  • Content that is lengthy or complex, which may overwhelm users in a modal
  • Multiple actions or options that require extensive user input
  • Navigation or content that can be better served with a dedicated page or section
  • On mobile devices if the content could be a full-page view

Best Practices for Progress Wizard

Progress wizards are useful for guiding users through multi-step processes. Here are some best practices to ensure accessibility:

  • Semantic HTML: Use (ol) for the step list and (li) for each step, for ARIA Attributes: use aria-current="step" for current step, aria-disabled="true" for inactive steps
  • Keyboard Navigation: Ensure all interactive elements are tabbable with proper focus styles
  • Screen Reader Support: Provide clear status updates when steps change
  • Clear Progress Indicator: Ensure visual connection between steps (lines, arrows, or connectors), maintain step states by ensuring distinct styles for completed, current, and upcoming steps. Ensure responsive layout, that is horizontal on desktop, vertical or compressed on mobile and abide with the colour contrast minimum 4.5:1 for text and interactive elements
  • Validation: Prevent advancing until current step is complete, and provide feedback, by showing clear error messages for validation failures
  • Persistence: Save progress if user leaves and returns, and allow returning to previous steps when appropriate
  • Clear Labels: Ensure concise step titles with optional descriptions, show current position (e.g., "Step 2 of 5"), and maintain consistent button placement (typically bottom-right for next, bottom-left for back)

  • Three-Dimensional Wizard: Z-Axis Navigation, that is steps appear to move forward/backward in 3D space, and visual cues showing progress through z-space
  • Ambient Wizard: Background Progression, that is, availability of a subtle, always-visible progress indicator, gentle notifications when steps complete, and contextual help. Help appears based on current step

Best Practices for Pagination

Pagination is useful for displaying large sets of data. Here are some best practices to ensure accessibility:

  • Clear Navigation: Use numbered pages, "Next" and "Previous" buttons, and ensure users can easily jump to specific pages
  • Consistent Layout: Keep pagination controls in a predictable location, usually at the bottom of the content
  • Mobile-Friendly Design: Ensure pagination works well on smaller screens, using touch-friendly buttons
  • Logical Page Limits: Avoid overwhelming users with too many items per page‚Äî20 to 50 items per page is a common range
  • Accessible Controls: Ensure pagination is keyboard-navigable and screen-reader friendly, using proper ARIA attributes
  • Self-Canonicalization: Each paginated page should have a canonical tag pointing to itself to prevent duplicate content issues
  • Internal Linking: Ensure paginated pages link to each other logically to distribute link equity
  • Avoid Infinite Scroll for SEO: If using infinite scroll, provide a fallback pagination system so search engines can index all content
  • Lazy Loading: Consider lazy loading for images and content to improve performance on paginated pages
  • Efficient Database Queries: Optimize queries to fetch only the necessary data per page
  • Cache Pagination Data: Reduce server load by caching frequently accessed paginated content
  • Descriptive URLs: Use meaningful URLs like example.com/products?page=2 instead of cryptic query strings

Best Practices for Sorting

    Best Practices for Loading Spinner

    Loading spinners are useful for indicating loading states. Here are some best practices to ensure accessibility:

    When & Where to Use Spinners

    • Indicate system activity: Use spinners for processing tasks, such as API calls, content fetching, or form submissions
    • Avoid unnecessary spinners: If a page loads quickly (less than 1 second), skipping the spinner improves usability
    • Use for asynchronous operations: Background loading of elements benefits from subtle animation cues
    • Position smartly: Place the spinner near relevant content (e.g., inside a button for submission loading, or center-screen for page loads)

    Duration & Timing

    • Short tasks: For tasks under 1 second, consider using a progress bar instead of a spinner
    • Fast response: If the loading time is under 2 seconds, use simple animations
    • Delayed spinner: If loading exceeds 500ms, introduce a spinner to signal activity (instant spinners sometimes feel broken)
    • Skeleton loaders for long delays: For delays exceeding 3 seconds, skeleton screens (gray placeholders) are preferable over spinners for better perceived performance

    Accessibility Enhancements

    • Screen reader support: Use aria-live="polite" to announce loading status, and role="status" for screen readers
    • Focus management: Ensure focus is not trapped in the spinner, and allow users to escape using the Escape key
    • Keyboard navigation: Ensure spinners are keyboard-navigable, and provide clear focus indicators
    • Color contrast: Maintain a minimum contrast ratio of 4.5:1 for text and background colours
    • Animation control: Provide options to pause or stop animations for users with motion sensitivity
    • Progressive enhancement: Ensure basic functionality without JavaScript, and provide fallbacks for older browsers
    • Test with real users: Ensure the spinner design is understandable and accessible to your target audience

    Responsive Design Considerations

    • Fluid layouts: Ensure spinners adapt to different screen sizes and orientations
    • Mobile-friendly: Use smaller, touch-friendly spinners for mobile devices
    • Orientation changes: Ensure spinners adjust correctly when the device orientation changes
    • Reduced motion mode: Provide a static alternative for users who prefer prefers-reduced-motion
